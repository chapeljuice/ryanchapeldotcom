Dev interviews are hard

During my recent "unemployment" stint (even though I was still getting paid...long story) I had some time to connect with a few different companies and endure some phone interviews. They are THE. WORST.

To be completely honest, I think they can be somewhat unfair, and if you don't get a good interviewer or they choose a particular question that maybe you just don't remember off the top of your head perfectly you can go from "confident about the interview" to "wondering when it's going to be over", fast.

Below are a few technical questions I was asked during said phone interviews. I'll omit the companies they came from, but I hope this can help someone get prepared for what may come. Cheers! :)

~Course API Consumer~

We use an API driven architecture. We've created a very simplified
version of our api that just exposes some data about courses here:
[website removed so company remains anonymous, sorry!]

We'd like you to build a simple application that consumes this api
and displays the data to the user. You are free to use whatever technology
stack you'd like but please pick something you know; this isn't a good
time to experiment in new technology! :)

What we hope to see is a web page that displays a list of courses.
The api returns 2 courses at a time with pagination links in the
response header. We'd like your app to display things the same way.
So 2 courses per page with next/prev links to get additional pages of
courses. When you click on a course, we'd like to be taken to a page
that displays all the details about that course.

In order to access courses, you'll need to send an authorization token
with your requests. The documentation at the above heroku link explains
how to obtain a token.

If you have time, you could also put an "Enroll" button on the individual
course page, which could post to the enrollment endpoint.
